<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller attrs -> `Paddle.Transactions.create/2` | Untrusted caller input must be reduced to the strict Phase 4 body contract and rejected when required identifiers, items, or checkout nesting are malformed. |
| local transaction builder -> Paddle API request body | Only the supported hosted-checkout branch may cross the boundary; unsupported transaction branches must not leak into the public request body. |
| Paddle response payload -> `%Paddle.Transaction{}` | The SDK must locally unwrap `data` and convert nested checkout payloads into a safe dot-accessible struct without changing the shared transport layer. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | T | `Paddle.Transactions.create/2` request shaping | mitigate | Allow only `customer_id`, `address_id`, `items`, optional `custom_data`, optional `checkout.url`, and the internal `"collection_mode" => "automatic"` constant so unsupported transaction branches cannot leak into the API body. |
| T-04-06 | T | identifier and nested attr validation | mitigate | Reject blank `customer_id`, blank `address_id`, malformed recurring items, and malformed checkout nesting with exact local tuples before dispatch. |
| T-04-07 | T | response checkout mapping | mitigate | Hydrate `data["checkout"]` into `%Paddle.Transaction.Checkout{}` in a private helper so `transaction.checkout.url` remains reliable and no raw nested map escapes as the documented contract. |
| T-04-08 | R | public API stability | mitigate | Lock method, path, JSON body, local validation tuples, API-error behavior, and transport passthrough in adapter-backed tests. |
</threat_model>
