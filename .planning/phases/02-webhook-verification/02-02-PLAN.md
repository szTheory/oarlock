<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| raw webhook request -> `verify_signature/4` | Untrusted header and body bytes must be parsed and verified without transformation or assumptions. |
| parsed signature header -> HMAC comparison | Candidate digests and timestamps must be validated before comparison to avoid malformed input and replay acceptance. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-05 | S | `Paddle.Webhooks.verify_signature/4` | mitigate | Recompute HMAC-SHA256 from the exact `"{ts}:{raw_body}"` payload using the shared secret and only accept a matching `h1` digest. |
| T-02-06 | R | `Paddle.Webhooks.verify_signature/4` | mitigate | Enforce a bidirectional timestamp tolerance with default `5` seconds and `opts[:now]`-driven tests to block replay and future-dated acceptance. |
| T-02-07 | T | Signature header parser | mitigate | Fail closed on malformed segments, missing `ts`, missing `h1`, empty `h1`, non-integer `ts`, and malformed digest lengths before comparison. |
| T-02-08 | I | Digest comparison path | mitigate | Use constant-time comparison on normalized equal-length hex digests to reduce timing side-channel leakage. |
| T-02-09 | D | Verification surface | accept | Pure local verification can still be called repeatedly by a consumer, but the SDK adds no network or global-state amplification in this phase. |
</threat_model>
